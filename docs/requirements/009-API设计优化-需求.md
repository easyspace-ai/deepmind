# 需求 009: API 设计优化

## 背景

随着 API 接口增多，代码风格出现了不一致，需要进行统一优化，提升可维护性。

## 需求列表

### 1. Handler 方法风格统一

**现状**
文件 `internal/api/handler.go` 和 `internal/api/channel_handler.go` 中存在两种风格：

风格 A - 使用 switch 处理多方法:
```go
func (h *Handler) handleChannels(c *gin.Context) {
    switch c.Request.Method {
    case http.MethodGet:
        h.listChannels(c)
    case http.MethodPost:
        h.createChannel(c)
    }
}
```

风格 B - 单独的方法:
```go
func (h *Handler) updateChannel(c *gin.Context) { ... }
func (h *Handler) deleteChannel(c *gin.Context) { ... }
```

**问题**
- 风格不统一，维护困难
- 路由注册分散，不够清晰

**需求**
- 统一使用风格 B（单独方法）
- 路由注册集中管理
- 方法命名遵循 `动词+资源` 规范

**验收标准**
- [ ] 重构 `handleChannels` -> `listChannels` + `createChannel`
- [ ] 重构 `handleChannelByID` -> `getChannel` + `updateChannel` + `deleteChannel`
- [ ] 重构 `handleAgents`, `handleAgentByID` 等类似方法
- [ ] 更新路由注册代码
- [ ] 确保所有测试通过

---

### 2. 错误响应格式统一

**现状**
项目中存在多种错误响应格式：

格式 A - gin.H:
```go
c.JSON(http.StatusBadRequest, gin.H{"error": "invalid request body"})
```

格式 B - ErrorResponse:
```go
c.JSON(http.StatusBadRequest, ErrorResponse{Error: "invalid id"})
```

格式 C - 内联结构体:
```go
c.JSON(http.StatusBadRequest, struct{ Error string }{Error: "..."})
```

**需求**
- 统一使用 `ErrorResponse` 结构体
- 添加错误码支持（可选）
- 统一错误消息语言（中文/英文）

**验收标准**
- [ ] 将所有 `gin.H{"error": ...}` 替换为 `ErrorResponse{...}`
- [ ] 统一错误消息语言风格
- [ ] 创建错误码常量定义（可选）
- [ ] 前端相应调整以适配统一格式

---

### 3. 响应结构统一化

**现状**
- `ListResponse` 结构在不同接口中字段名不一致
- 有些接口返回 `items`，有些返回 `data`
- 分页字段有时用 `page/pageSize`，有时用 `offset/limit`

**需求**
- 统一 List 响应结构:
  ```go
  type ListResponse struct {
      Items    interface{} `json:"items"`
      Total    int64       `json:"total"`
      Page     int         `json:"page"`
      PageSize int         `json:"page_size"`
  }
  ```
- 统一 Single 响应结构（使用包装器）:
  ```go
  type DataResponse struct {
      Data interface{} `json:"data"`
  }
  ```

**验收标准**
- [ ] 统一所有 List 接口返回 `ListResponse`
- [ ] 统一所有 Single 接口返回 `DataResponse`
- [ ] 更新前端类型定义
- [ ] 确保前端正确处理新格式

---

### 4. 请求参数校验统一

**现状**
- 部分接口使用 `ShouldBindJSON` 后直接处理
- 缺少统一的请求参数校验逻辑
- 错误信息返回不够友好

**需求**
- 引入请求参数校验中间件或使用 validation tags
- 统一的校验错误返回格式
- 支持字段级错误提示

**验收标准**
- [ ] 在 Model/Request 结构体中添加 validation tags
- [ ] 创建统一的校验错误处理器
- [ ] 返回友好的字段级错误信息

## 优先级

| 需求 | 优先级 | 原因 |
|------|--------|------|
| 错误响应格式统一 | P1 | 影响前端一致性 |
| Handler 方法统一 | P2 | 代码可维护性 |
| 响应结构统一 | P2 | API 一致性 |
| 请求参数校验 | P2 | 健壮性提升 |
