# 1. 背景（Why）

当前系统的Task管理功能存在以下问题：
1. **无实时更新**：前端必须手动刷新才能看到新创建的任务或状态变化
2. **无法手动创建**：用户只能依赖AI Agent创建任务，不能在前端直接发起
3. **缺少管理功能**：无法筛选、搜索任务，缺少执行时长、进度等信息展示
4. **状态通知不足**：任务完成/失败时没有前端通知，用户容易错过

为了提升Task管理的用户体验，需要对Task管理界面进行全面重构，支持WebSocket实时推送、手动创建任务、增强的筛选和通知功能。

# 2. 目标（What，必须可验证）

- [ ] 扩展WebSocket协议，支持Task相关消息类型（创建、更新、完成）
- [ ] 后端Task Manager在状态变化时主动推送WebSocket消息
- [ ] 前端Task列表实时更新（新任务、状态变化、执行日志）
- [ ] 前端支持手动创建Task（弹出表单输入任务内容）
- [ ] 前端支持任务筛选（按状态、时间范围、关键词搜索）
- [ ] 前端任务完成/失败时显示通知提醒
- [ ] 优化任务详情展示（执行时长、实时日志流）
- [ ] 支持任务重试功能（对失败/已停止的任务）

# 3. 非目标（Explicitly Out of Scope）

- 不支持修改已创建任务的内容
- 不支持任务的暂停/恢复（只能停止）
- 不支持任务的优先级调整
- 不实现任务的批量操作（批量停止、删除）
- 不替代现有的AI Agent任务创建方式（两者共存）
- 不修改现有的Task持久化存储格式（保持YAML文件存储）

# 4. 使用场景 / 用户路径

## 场景 1：用户手动创建后台任务

1. 用户登录系统
2. 进入"后台任务"页面
3. 点击"新建任务"按钮
4. 在弹出的表单中输入任务内容描述
5. 点击确认，任务立即创建并开始执行
6. 页面实时显示新任务出现在列表顶部（状态：运行中）

## 场景 2：用户监控任务执行情况

1. 用户打开"后台任务"页面
2. 页面自动连接WebSocket
3. 用户看到当前所有运行中/已完成的任务
4. 当某个任务状态变化时（运行中→完成/失败），列表自动更新
5. 任务完成时，页面右上角显示通知提示
6. 用户点击任务"详情"查看执行日志和结果

## 场景 3：用户搜索和筛选任务

1. 用户在任务列表顶部的搜索框输入关键词
2. 列表实时过滤显示匹配的任务（按任务内容匹配）
3. 用户选择状态筛选（如只显示"运行中"）
4. 列表只显示运行中的任务
5. 用户选择时间范围（今天/最近7天/全部）
6. 列表按时间范围过滤

## 场景 4：用户停止运行中的任务

1. 用户在任务列表中找到运行中的任务
2. 点击"停止"按钮
3. 确认停止操作
4. 任务状态立即更新为"已停止"
5. WebSocket推送状态变化到前端

## 场景 5：用户重试失败的任务

1. 用户在任务列表中看到失败/已停止的任务
2. 点击"重试"按钮
3. 系统创建一个新任务，使用相同的任务内容
4. 新任务出现在列表顶部，开始执行

# 6. 约束条件

## 技术约束

- WebSocket消息格式必须为JSON
- 使用现有的WebSocket连接，不新建独立连接
- 向后兼容：未升级的前端仍能正常使用（基于HTTP轮询）
- 前端使用Ant Design组件库

## 架构约束

- Task Manager必须通过现有的MessageBus发送WebSocket消息
- 必须复用现有的Task模型和存储机制
- WebSocket Handler必须支持消息类型路由

## 安全约束

- 手动创建任务需要用户登录（JWT验证）
- 用户只能看到自己创建的任务（管理员可以看到全部）
- 停止/重试操作只能由任务创建者或管理员执行

## 性能约束

- WebSocket消息推送延迟 < 100ms
- 任务列表支持虚拟滚动（如果任务数量超过1000）
- 日志流支持增量加载，不一次性加载全部

# 7. 可修改 / 不可修改项

- ❌ 不可修改：
  - 现有的Task状态机（pending/running/finished/failed/stopped）
  - 现有的Task持久化存储格式（YAML文件）
  - 现有的AI Agent任务创建流程
  - 现有的WebSocket连接建立方式（URL、认证方式）

- ✅ 可调整：
  - WebSocket消息协议中的字段命名
  - 前端界面布局和样式
  - 筛选条件的具体选项
  - 通知展示方式（右上角/页面内）

# 8. 接口与数据约定

## 新增HTTP接口

### 手动创建任务

```
POST /api/v1/tasks
```

请求体：
```json
{
  "work": "任务内容描述"
}
```

响应：
```json
{
  "id": "000123",
  "status": "running",
  "work": "任务内容描述",
  "created_at": "2026-03-19T10:00:00Z"
}
```

### 重试任务

```
POST /api/v1/tasks/:id/retry
```

响应：
```json
{
  "id": "000124",
  "status": "running",
  "work": "任务内容描述（复制自原任务）",
  "created_at": "2026-03-19T10:05:00Z"
}
```

### 扩展任务列表接口（新增查询参数）

```
GET /api/v1/tasks?status=running,failed&since=2026-03-18&keyword=搜索关键词
```

查询参数：
- `status`: 状态筛选，逗号分隔（可选）
- `since`: 起始时间，ISO8601格式（可选）
- `until`: 结束时间，ISO8601格式（可选）
- `keyword`: 关键词搜索，匹配任务内容（可选）
- `creator`: 创建者用户ID（可选，管理员可用）

## WebSocket消息协议扩展

### 任务创建消息（服务端→客户端）

```json
{
  "type": "task_created",
  "payload": {
    "id": "000123",
    "status": "running",
    "work": "任务内容",
    "channel": "websocket",
    "chat_id": "chat_xxx",
    "created_at": "2026-03-19T10:00:00Z",
    "created_by": "user_xxx"
  },
  "timestamp": 1710000000000
}
```

### 任务更新消息（服务端→客户端）

```json
{
  "type": "task_updated",
  "payload": {
    "id": "000123",
    "status": "running",
    "result": "部分执行结果...",
    "logs": ["日志1", "日志2"],
    "updated_at": "2026-03-19T10:01:00Z"
  },
  "timestamp": 1710000000000
}
```

### 任务完成消息（服务端→客户端）

```json
{
  "type": "task_completed",
  "payload": {
    "id": "000123",
    "status": "finished",
    "result": "完整执行结果...",
    "logs": ["日志1", "日志2", "日志3"],
    "completed_at": "2026-03-19T10:05:00Z",
    "duration_seconds": 300
  },
  "timestamp": 1710000000000
}
```

状态枚举：`finished` | `failed` | `stopped`

### 任务日志消息（可选，用于实时日志流）

```json
{
  "type": "task_log",
  "payload": {
    "id": "000123",
    "log": "新增日志行内容",
    "timestamp": "2026-03-19T10:02:00Z"
  },
  "timestamp": 1710000000000
}
```

# 9. 验收标准（Acceptance Criteria）

- **AC1**: 如果用户在前端点击"新建任务"并输入内容，那么新任务立即出现在列表顶部，且状态为"运行中"
- **AC2**: 如果AI Agent创建了任务，那么前端无需刷新即可实时看到新任务出现
- **AC3**: 如果任务状态发生变化（如运行中→完成），那么前端列表自动更新，并显示通知
- **AC4**: 如果用户在搜索框输入关键词，那么列表只显示任务内容匹配的任务
- **AC5**: 如果用户选择状态筛选，那么列表只显示对应状态的任务
- **AC6**: 如果用户点击"停止"运行中的任务，那么任务状态立即变为"已停止"
- **AC7**: 如果用户点击"重试"失败的任务，那么创建一个新任务并使用相同内容
- **AC8**: 如果用户打开任务详情，可以看到完整的执行日志和结果
- **AC9**: 如果WebSocket连接断开，前端自动重连，且重连后能继续接收任务更新
- **AC10**: 如果任务完成或失败，页面右上角显示通知提醒

# 10. 风险与已知不确定点

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| WebSocket消息过多导致前端性能问题 | 页面卡顿 | 实现消息节流/防抖，控制更新频率 |
| 任务数量过多导致列表渲染缓慢 | 用户体验差 | 实现虚拟滚动或分页加载 |
| 向后兼容性问题 | 旧版本前端无法使用 | 保持HTTP API完整，WebSocket为增量功能 |
| 权限控制不完善 | 用户看到/操作他人任务 | 后端严格校验权限 |

## 不确定点

- 是否需要支持任务执行进度百分比（需要Task Manager提供进度信息）
- 实时日志流的频率控制策略
- 通知展示的具体UI设计（全局通知 vs 页面内通知）
