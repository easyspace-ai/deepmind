# 需求 011: 前端优化

## 背景

前端代码随着功能增加，存在一些可优化的地方，需要提升代码质量和用户体验。

## 需求列表

### 1. API 响应类型与后端对齐

**现状**
文件: `web/src/types/index.ts`
- 部分类型定义与后端 API 响应不完全一致
- 缺少某些新增字段（如 Code 字段）

**问题示例**
后端返回 `user_code`，前端类型可能定义为 `userCode`。

**需求**
- 统一前后端字段命名（snake_case vs camelCase）
- 完善所有类型的字段定义
- 添加类型守卫函数（type guards）

**验收标准**
- [ ] 审查所有类型定义与后端 API 的匹配度
- [ ] 统一字段命名规范（建议前端使用 camelCase，转换层处理）
- [ ] 添加必要的类型守卫
- [ ] 消除 `any` 类型的使用

---

### 2. API 客户端统一错误处理

**现状**
文件: `web/src/api/client.ts`（假设）
- 错误处理分散在各 API 调用中
- 401 未认证情况需要统一处理

**需求**
- 添加请求/响应拦截器
- 统一处理 401 错误（自动跳转登录）
- 统一的错误提示

**验收标准**
- [ ] 实现 axios/fetch 拦截器
- [ ] 401 错误自动清除 token 并跳转登录页
- [ ] 500 错误显示友好的错误提示
- [ ] 网络错误处理

---

### 3. 前端路由权限控制

**现状**
文件: `web/src/App.tsx`
- 路由定义集中
- 缺少权限控制

**需求**
- 基于角色的路由权限控制
- 未登录用户自动跳转登录页
- 登录后返回原页面

**验收标准**
- [ ] 创建路由守卫组件
- [ ] 需要登录的路由自动检查认证状态
- [ ] 未认证时跳转登录页并记录原页面
- [ ] 登录后自动返回原页面

---

### 4. Token 存储安全优化

**现状**
文件: `web/src/api/auth.ts`
- Token 存储在 localStorage
- 存在 XSS 攻击风险

**需求**
- 考虑使用 httpOnly Cookie（需要后端支持）
- 或实现 Token 刷新机制
- 添加 Token 过期检测

**验收标准**
- [ ] 实现 Token 过期前自动刷新（可选）
- [ ] 检测 Token 过期并自动跳转登录
- [ ] 评估迁移到 httpOnly Cookie 的可行性

---

### 5. API 层代码生成调研

**现状**
- 手写 API 客户端代码
- 容易与后端 API 不同步

**需求**
- 调研 API 代码生成方案
- OpenAPI/Swagger 规范
- 前端代码自动生成

**方案对比**
| 方案 | 优点 | 缺点 |
|------|------|------|
| 手写维护 | 灵活 | 容易不同步 |
| OpenAPI Generator | 自动生成 | 需要维护 OpenAPI 规范 |
| orval + MSW | 生成 + Mock | 工具链复杂 |

**验收标准**
- [ ] 调研 OpenAPI 方案的可行性
- [ ] 评估成本收益
- [ ] 如可行，创建实施方案

## 优先级

| 需求 | 优先级 | 原因 |
|------|--------|------|
| 统一错误处理 | P1 | 用户体验关键 |
| 类型对齐 | P1 | 减少运行时错误 |
| 路由权限控制 | P2 | 安全性提升 |
| Token 安全 | P2 | 安全考虑 |
| API 代码生成 | P3 | 长期维护性 |
