## 1. 需求背景

当前系统中，当用户与 Agent 进行长时间对话时，如果对话陷入死循环或执行时间过长，用户无法主动停止该对话。这会导致：
- 用户必须等待对话自然结束或超时
- 系统资源被长时间占用
- 用户体验不佳

## 2. 需求目标

实现对单个会话（Session）的强制取消功能，允许用户主动停止正在进行的对话任务。

## 3. 功能需求

### 3.1 核心功能

1. **会话级 Context 管理**
   - 为每个活跃的会话创建独立的 cancellable context
   - 将 context 的 cancel 函数与会话关联存储

2. **取消会话接口**
   - 提供 API 接口用于取消指定会话
   - 接口接收 session_key 作为参数
   - 调用对应会话的 cancel 函数

3. **取消后的处理**
   - 正在执行的 ADK Runner 应感知 context 取消并停止
   - 向用户返回"对话已取消"的提示
   - 清理会话相关的资源

### 3.2 非功能需求

- 取消操作应能快速响应（< 100ms）
- 不影响其他会话的正常运行
- 取消后的会话可以重新开始新对话

## 4. 接口设计

### 4.1 API 接口

```
POST /api/v1/sessions/{session_key}/cancel

响应：
{
  "success": true,
  "message": "会话已取消"
}
```

## 5. 影响范围

- `session/manager.go` - 会话管理器，需要存储 cancel 函数
- `agent/run.go` - Loop.Run，需要为每条消息创建独立 context
- `agent/interruptible.go` - 需要传播 context 取消信号
- `internal/api/` - 需要新增 API 接口

## 6. 风险与挑战

1. **Context 传播** - 确保 context 能正确传递到 ADK Runner
2. **资源清理** - 取消后需要正确清理资源，避免内存泄漏
3. **并发安全** - 多个 goroutine 访问 session 的 cancel 函数需要加锁

## 7. 验收标准

- [ ] 可以正常取消正在执行的会话
- [ ] 取消后会话立即停止，不再产生新的 AI 响应
- [ ] 取消后用户可以继续发送新消息开始新对话
- [ ] 不影响其他会话的正常运行
