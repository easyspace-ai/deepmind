## 1. 需求背景

当前系统使用数据库自增 ID（如 user_id、channel_id、agent_id）作为业务实体的唯一标识。这种方案在单机环境下工作良好，但存在以下问题：

1. **分布式扩展困难**：自增 ID 在分布式环境下容易冲突
2. **数据迁移复杂**：ID 依赖数据库生成，跨实例迁移时难以保持关系
3. **安全性问题**：连续整数 ID 容易被遍历攻击
4. **多租户隔离不足**：ID 本身不包含租户信息

为了支持未来大规模部署、分布式架构和多实例部署，需要引入业务编码系统。

---

## 2. 需求目标

### 2.1 核心目标

为以下业务实体添加 **Code** 字段作为业务唯一标识：

| 实体      | Code 字段名      | 说明           |
|-----------|------------------|----------------|
| User      | user_code        | 用户编码       |
| Channel   | channel_code     | 渠道编码       |
| Agent     | agent_code       | Agent 编码     |
| Provider  | provider_code    | 提供商编码     |
| CronJob   | cron_job_code    | 定时任务编码   |

### 2.2 Code 生成规则

1. **格式**：短字符串，8-16 位字符
2. **字符集**：大小写字母 + 数字（去除易混淆字符如 0/O, 1/I/l）
3. **唯一性**：全局唯一，不区分大小写
4. **可读性**：避免连续相同字符，提高可读性
5. **示例**：`usr_a3x9k2m`, `chn_7pq4w8z`, `agt_k9m2p5x`

### 2.3 功能需求

#### FR-001：Code 自动生成
- 创建实体时，系统自动生成 Code
- Code 一旦生成不可修改

#### FR-002：API 支持 Code 查询
- 支持通过 Code 查询实体详情
- 支持 Code 作为关联字段进行查询

#### FR-003：业务交互使用 Code
- 对外 API 优先使用 Code 而非 ID
- 前端界面展示 Code
- 日志记录使用 Code

#### FR-004：向后兼容
- 现有 ID 机制继续保留
- 内部关联仍可使用 ID（性能考虑）
- 新功能优先使用 Code

#### FR-005：Code 索引
- Code 字段需建立唯一索引
- 支持 Code 的快速查询

---

## 3. 非功能需求

### 3.1 性能要求
- Code 生成算法需保证高吞吐量（>1000/s）
- Code 查询性能应与 ID 查询相当

### 3.2 安全要求
- Code 不可被预测
- Code 生成需考虑防碰撞

### 3.3 扩展性
- 支持未来添加新的实体类型
- 支持自定义 Code 前缀规则

---

## 4. 边界与限制

### 4.1 范围限定
- 本期仅实现 User、Channel、Agent 三个实体的 Code 系统
- Provider、CronJob 作为二期扩展

### 4.2 不实现功能
- 不支持用户自定义 Code
- 不支持 Code 修改/回收

| 编号 | 验收项 | 验收标准 |
|------|--------|----------|
| AC-001 | Code 生成 | 创建用户/渠道/Agent 时自动生成 Code |
| AC-002 | 唯一性 | 相同类型的 Code 全局唯一 |
| AC-003 | API 查询 | 可通过 Code 查询实体详情 |
| AC-004 | 日志记录 | 关键操作日志使用 Code 标识实体 |
| AC-005 | 性能测试 | Code 生成和查询性能满足要求 |
