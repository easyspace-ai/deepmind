## 1. 背景（Why）

当前系统已有完整的 Task 后台任务管理功能（`pkg/agent/task`），Agent 可以通过工具接口启动后台任务执行异步工作。但是这些任务没有前端管理界面，管理员无法直观地查看任务状态、日志或停止任务。

## 2. 目标（What，必须可验证）

- [ ] 管理员可以通过 Web UI 查看所有任务列表（运行中 + 今日已完成）
- [ ] 管理员可以查看单个任务的详细信息（状态、日志、结果）
- [ ] 管理员可以停止正在运行的任务
- [ ] 任务状态实时刷新（手动刷新）

## 3. 非目标（Explicitly Out of Scope）

- 不实现通过 Web UI 创建新任务（任务只能由 Agent 通过工具创建）
- 不实现 WebSocket 实时推送（使用手动刷新）
- 不实现历史任务查询（只显示今日任务）
- 不实现任务编辑功能

## 4. 使用场景 / 用户路径

### 场景1：查看任务列表

1. 管理员登录系统
2. 点击左侧菜单"后台任务"
3. 系统显示所有任务列表，包括运行中和今日已完成的任务
4. 列表显示：任务ID、状态、创建时间、渠道、任务内容摘要

### 场景2：查看任务详情

1. 在任务列表中点击"查看详情"按钮
2. 弹窗显示任务完整信息：
   - 基本信息：ID、状态、创建时间、渠道、ChatID
   - 任务内容：完整的 work 描述
   - 执行日志：最近 N 条日志
   - 执行结果：任务完成后的结果摘要

### 场景3：停止运行中的任务

1. 在任务列表中找到运行中的任务
2. 点击"停止"按钮
3. 系统确认后停止任务
4. 任务状态变为"已停止"

## 5. 功能需求清单（Checklist）

- [ ] 后端 API：获取任务列表 `GET /api/v1/tasks`
- [ ] 后端 API：获取单个任务详情 `GET /api/v1/tasks/:id`
- [ ] 后端 API：停止任务 `POST /api/v1/tasks/:id/stop`
- [ ] 前端页面：任务列表展示
- [ ] 前端页面：任务详情弹窗
- [ ] 前端页面：停止任务功能
- [ ] 前端页面：刷新任务列表

## 6. 约束条件

### 技术约束

- 后端使用现有的 `pkg/agent/task.Manager` 实现
- 前端使用 React + TypeScript + Ant Design
- API 遵循现有的 RESTful 风格

### 架构约束

- Task Service 层包装 task.Manager，提供 API 所需接口
- 前端页面结构与 Skills 管理页面保持一致

### 安全约束

- 所有 API 需要登录认证（复用现有中间件）
- 停止任务操作需要确认

### 性能约束

- 任务列表加载时间 < 2s
- 任务详情加载时间 < 1s

## 7. 可修改 / 不可修改项

- ❌ 不可修改：`pkg/agent/task` 包的核心逻辑
- ❌ 不可修改：现有的数据结构（`Info`、`PersistedTask`）
- ✅ 可调整：前端 UI 布局和样式
- ✅ 可调整：API 响应格式（在合理范围内）

## 8. 接口与数据约定

### API 定义

#### GET /api/v1/tasks

获取任务列表

**响应**：
```json
{
  "items": [
    {
      "id": "000001",
      "status": "running",
      "work": "分析代码并生成报告",
      "channel": "wechat",
      "chat_id": "chat_123",
      "created_at": "2026-03-17T10:00:00Z",
      "completed_at": null,
      "result": ""
    }
  ],
  "total": 1
}
```

#### GET /api/v1/tasks/:id

获取任务详情

**响应**：
```json
{
  "id": "000001",
  "status": "running",
  "work": "分析代码并生成报告",
  "channel": "wechat",
  "chat_id": "chat_123",
  "created_at": "2026-03-17T10:00:00Z",
  "completed_at": null,
  "result": "",
  "logs": ["任务已创建", "任务启动", "正在分析..."]
}
```

#### POST /api/v1/tasks/:id/stop

停止任务

**响应**：
```json
{
  "message": "任务已停止",
  "data": {
    "id": "000001",
    "status": "stopped"
  }
}
```

### 状态枚举

| 状态 | 含义 |
|------|------|
| `pending` | 等待中 |
| `running` | 运行中 |
| `finished` | 已完成 |
| `failed` | 失败 |
| `stopped` | 已停止 |

## 9. 验收标准（Acceptance Criteria）

1. 如果管理员访问后台任务页面，则能看到所有任务列表
2. 如果任务状态为运行中，则显示"停止"按钮
3. 如果点击"停止"按钮并确认，则任务状态变为"已停止"
4. 如果点击"查看详情"按钮，则弹窗显示任务完整信息
5. 如果点击刷新按钮，则重新加载任务列表
6. 页面风格与现有管理页面保持一致

## 10. 风险与已知不确定点

- 任务日志容量限制（默认保留最近 10 条），超出部分不显示
- 任务持久化存储在 YAML 文件中，大量任务时可能有性能问题（当前设计中只显示今日任务）

## 11. 非目标

- 不实现任务的 CRUD 完整操作（只能查看和停止）
- 不实现任务调度功能（由 Agent 控制）
- 不实现任务重试功能
