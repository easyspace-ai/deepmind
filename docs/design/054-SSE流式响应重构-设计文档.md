## 1. 背景与目标

### 1.1 背景
- 当前项目 `langgraph_handler.go` 中的 SSE 实现存在问题：
  - ❌ 流式响应不显示（前端收不到 SSE 事件）
  - ❌ 连接不稳定（经常断开或超时）
- 项目中已有两个参考实现：
  - `test/Agent-Eino/internal/handler/sse.go` - eino 简洁实现
  - `test/aggo/pkg/sse/` - 完善的独立 SSE 包（MIT 许可证）

### 1.2 目标
- 修复 SSE 流式响应问题，确保前端能正常接收事件
- 确保连接稳定，正确处理客户端断开
- 复用成熟的第三方实现，减少自行维护成本
- 对齐 `frontend/` 当前使用的 LangGraph SDK 流式协议，保证前后端联调可用

### 1.3 非目标
- 不修改 DeerFlow 的事件格式协议（保持 `messages-tuple`/`values`/`end`）
- 不修改 `test/aggo/` 原项目（只复制使用）
- 不在本次改动中实现完整的 `joinStream` 恢复机制

## 2. 问题分析

### 2.1 当前实现的问题
```go
// internal/api/langgraph_handler.go - 当前实现
sendEvent := func(event string, data interface{}) {
    dataBytes, _ := json.Marshal(data)
    fmt.Fprintf(w, "event: %s\n", event)
    fmt.Fprintf(w, "data: %s\n\n", dataBytes)
    flusher.Flush()
}
```

问题：
1. 手动格式化 SSE 协议，容易出错
2. 没有完善的连接管理
3. 没有对 `flusher` 的可用性做充分检查
4. 流式读取逻辑复杂，有回退模拟流式逻辑
5. keep-alive goroutine 与主流程并发写同一个 `ResponseWriter`，容易导致 SSE 数据交错或连接异常
6. `stopKeepAlive` 存在重复关闭风险，可能触发 panic
7. 当前实际事件名为 `metadata`/`updates`/`finish`，与 DeerFlow 和 `frontend/` 主链路期望的 `messages-tuple`/`values`/`end` 不一致
8. `ChatModelAdapter.Stream()` 目前会先同步执行 `Generate()` 再回放整段消息，模型慢时 SSE 会长时间只有首包，无后续事件

### 2.2 frontend 联调现状

- `frontend/` 使用 `@langchain/langgraph-sdk` 的 `useStream` 和 `thread.submit()` 直接与 `/api/langgraph` 通信
- 页面层主要消费高层状态：
  - `thread.messages`
  - `thread.values.title`
  - `thread.values.todos`
  - `thread.values.artifacts`
- 当前前端已临时兼容 `updates` 流模式，但这只是过渡，不应继续扩大兼容面
- 前端当前开启了 `streamResumable`，但后端 `joinStream` 尚未实现，存在“前端以为可恢复、后端实际不可恢复”的错配

### 2.3 参考实现的优点

**aggo sse 包**：
- ✅ 完善的 `Writer` 抽象，封装 SSE 协议细节
- ✅ `Event` 对象池，减少内存分配
- ✅ 正确处理多行 data
- ✅ 连接状态管理（`IsClosed()`）

**eino sse 实现**：
- ✅ 简洁的 `streamMessageReader`，buffer 累积 delta
- ✅ 使用 `c.SSEvent()`  gin 内置方法
- ✅ 实时 flush 策略

### 4.2 pkg/sse/deerflow/ - DeerFlow 扩展

```go
// pkg/sse/deerflow/writer.go

package deerflow

import (
    "net/http"
    "github.com/weibaohui/nanobot-go/pkg/sse"
    "github.com/bytedance/sonic"
)

// Writer DeerFlow 专用 SSE Writer
// 封装 DeerFlow 特定的三种事件：
//  - messages-tuple: 消息增量更新
//  - values: 完整状态快照
//  - end: 流结束
type Writer struct {
    base *sse.Writer
}

// NewWriter 创建 DeerFlow SSE Writer
func NewWriter(id string, w http.ResponseWriter) *Writer {
    base := sse.NewWriter(id, w)
    if base == nil {
        return nil
    }
    return &Writer{base: base}
}

// WriteMessagesTuple 发送 messages-tuple 事件
// 用于发送 AI 消息的增量更新
func (w *Writer) WriteMessagesTuple(msg interface{}) error {
    return w.base.WriteEventJSON("", "messages-tuple", msg)
}

// WriteValues 发送 values 事件
// 用于发送完整的 thread 状态快照
func (w *Writer) WriteValues(values map[string]interface{}) error {
    return w.base.WriteEventJSON("", "values", values)
}

// WriteEnd 发送 end 事件
// 用于标记流结束
func (w *Writer) WriteEnd(data map[string]interface{}) error {
    return w.base.WriteEventJSON("", "end", data)
}

// Close 关闭 Writer
func (w *Writer) Close() {
    w.base.Close()
}

// IsClosed 检查是否已关闭
func (w *Writer) IsClosed() bool {
    return w.base.IsClosed()
}

// SetDone 设置完成回调
func (w *Writer) SetDone(f func(http.ResponseWriter) error) {
    w.base.SetDone(f)
}
```

### 4.3 langgraph_handler.go 重构

#### 4.3.1 主要改动
1. **删除** 当前复杂的 `sendEvent` 闭包
2. **使用** `deerflow.Writer` 替代
3. **简化** 流式读取逻辑，参考 eino 的 buffer 累积 + delta 追加模式
4. **移除** 回退模拟流式逻辑（强制真实流式，失败就报错）
5. **消除** keep-alive 并发写和重复 close 的稳定性问题
6. **统一** 结束路径，避免重复发送结束事件
7. **恢复** DeerFlow 标准事件名，优先保证 `frontend/` 联调

#### 4.3.2 frontend 联调调整

1. `frontend/src/core/api/stream-mode.ts` 保留对历史 `updates` 的兜底，但默认走标准模式
2. `frontend/src/core/threads/hooks.ts` 暂时关闭 `streamResumable`
3. 等后端实现 `joinStream` 后，再重新开启恢复能力

#### 4.3.3 重构后的核心流程

```
1. 设置 SSE 响应头
2. 创建 deerflow.Writer
3. 发送用户消息 (messages-tuple)
4. 获取 eino StreamReader
5. Loop:
   - 读取 chunk
   - buffer += chunk.Content (delta 追加，不是替换)
   - 达到 flush 阈值时:
     - 发送 messages-tuple 事件
     - 清空 buffer
6. 发送剩余 buffer
7. 更新 thread 状态
8. 发送 values 事件
9. 发送 end 事件
```

#### 4.3.4 关键修复点
- ✅ 使用 aggo 的成熟 `flusher.Flush()` 机制，确保数据真正发送
- ✅ 正确处理 context cancel，监听客户端断开
- ✅ 同一时刻只允许单线程写出 SSE 数据
- ✅ 事件协议对齐 `frontend/` 的 LangGraph SDK 预期
- ✅ 简化代码，减少出错可能
- ✅ 在模型调用期间发送安全 keepalive，避免首包后长时间静默
