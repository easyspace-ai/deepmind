## 核心问题

当前 nanobot-go 存在以下主要问题：

### 1️⃣ 架构问题

| 问题 | 影响 | 严重程度 |
|------|------|--------|
| **贫血模型** | 业务逻辑散落在 Service 层，难以维护 | 🔴 高 |
| **缺乏边界** | 模块之间耦合，难以扩展 | 🔴 高 |
| **隐式不变量** | 数据一致性无法保证 | 🟠 中 |
| **遗留器模式** | 接口设计不清晰 | 🟠 中 |

### 2️⃣ DeerFlow 集成差距

| 特性 | DeerFlow | nanobot-go | 差距 |
|------|----------|-----------|------|
| **提示词系统** | 模块化 Builder | 文件加载 | ❌ 缺失细粒度 |
| **沙箱系统** | 虚拟路径 + 验证 | 直接物理路径 | ❌ 缺失 |
| **中间件链** | 严格顺序 | 事件驱动 | ⚠️ 需优化 |
| **MCP 导入** | 系统级工具 | 用户级工具 | ⚠️ 架构错误 |

### 3️⃣ 代码质量

- 🔴 单元测试覆盖率 < 50%
- 🔴 无清晰的错误处理策略
- 🟠 日志不一致
- 🟠 缺乏集成测试

## 解决方案框架

### 🎯 目标

通过 **DDD（Domain-Driven Design）** 实现：

1. **清晰的边界** - 7 个 Bounded Context
2. **富领域模型** - 业务逻辑内聚到聚合根
3. **事件驱动** - 模块通过领域事件解耦
4. **可测试性** - 单元测试覆盖 ≥ 80%
5. **DeerFlow 最佳实践集成** - 完整功能对齐

### 📐 架构观

```
┌─────────────────────────────────────────────────────────────┐
│                    Application Layer (服务)                  │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  MessageProcessingService    AgentExecutorService    │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                          △
                          │ 依赖
┌─────────────────────────────────────────────────────────────┐
│                     Domain Layer (业务)                      │
│  ┌──────┐  ┌─────────┐  ┌─────────┐  ┌──────┐              │
│  │Agent │  │Session  │  │Message  │  │User  │  ...         │
│  │聚合  │  │聚合     │  │聚合     │  │聚合  │              │
│  └──────┘  └─────────┘  └─────────┘  └──────┘              │
│                                                              │
│            领域事件 (Event-Driven Decoupling)               │
│    CapabilityAdded → MessageAppended → ToolCallResolved    │
└─────────────────────────────────────────────────────────────┘
                          △
                          │ 实现
┌─────────────────────────────────────────────────────────────┐
│                Infrastructure Layer (实现)                   │
│  ┌─────────┐  ┌──────────┐  ┌─────────┐  ┌──────────┐      │
│  │Repositories│ EventBus  │  │Adapters │  │Sandbox   │      │
│  │(GORM)     │(Publisher)│  │(Channel)│  │(Sandbox) │      │
│  └─────────┘  └──────────┘  └─────────┘  └──────────┘      │
└─────────────────────────────────────────────────────────────┘
```

### 🔄 7 个 Bounded Contexts

| # | 上下文 | 聚合根 | 关键职责 |
|---|--------|--------|---------|
| 1 | **Agent Orchestration** | Agent | 身份、能力、配置 |
| 2 | **Session Management** | ConversationSession | 对话生命周期 |
| 3 | **Message & History** | ConversationRecord | 不可变消息历史 |
| 4 | **Channel Communication** | Channel | 多渠道适配 |
| 5 | **User & Auth** | User | 多租户隔离 |
| 6 | **LLM Provider** | LLMProvider | 模型配置访问 |
| 7 | **MCP Server** | MCPServer | 工具注册表 |

## 关键设计决定

### 1. 事件模型

**决定**: 使用 **事件发布模式**（不是完全 Event Sourcing）

```
优点:
✅ 简单易实现
✅ 足以支持解耦
✅ 没有过度工程

缺点:
❌ 无完整历史重放
❌ 无时间旅行调试
```

**替代方案**：如需完全 Event Sourcing，可在 Phase 4 升级。

### 2. 多租户隔离

**决定**: `UserCode` 字段作为所有聚合的租户标识

```go
// 强制多租户规则
type Agent struct {
    id   AgentID
    userCode string  // 🔑 租户键
    // ...
}

// 仓储必须检查
func (r *AgentRepository) FindByID(ctx, id) {
    // SELECT * FROM agents WHERE id = ? AND user_code = ctx.UserCode
}
```

### 3. 聚合大小

**决定**: 保持聚合小而专注

```
例如：
✅ ConversationSession 只包含消息 ID（不包含完整消息体）
✅ Agent 只包含能力列表（不包含能力详情）
❌ 不要把太多东西放入单个聚合
```

### 4. 值对象不可变性

**决定**: 通过 `unexported` 字段 + 方法接收器实现

```go
type Identity struct {
    name string  // 私有字段
    // ...
}

// 只有 New 函数和只读方法
func (i *Identity) Name() string { return i.name }

// ❌ 无 SetName() 方法
```
