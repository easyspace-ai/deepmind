## 1. 中间件钩子职责定义

### 1.1 钩子类型

基于 `pkg/agent/middleware/types.go`，中间件可以实现以下一个或多个接口：

| 接口 | 钩子方法 | DeerFlow 对应 | 职责 |
|------|---------|--------------|------|
| `BeforeAgentMiddleware` | `BeforeAgent()` | `before_agent()` | Agent 执行**前**，修改 state、注入消息、准备资源 |
| `BeforeModelMiddleware` | `BeforeModel()` | `before_model()` | 模型调用**前**，预处理消息 |
| `AfterModelMiddleware` | `AfterModel()` | `after_model()` | 模型响应**后**，检查/修改 AI 输出 |
| `WrapToolCallMiddleware` | `WrapToolCall()` | `wrap_tool_call()` | 工具调用**包装器**，拦截/转换工具调用 |
| `AfterAgentMiddleware` | `AfterAgent()` | `after_agent()` | Agent 执行**后**，清理资源、异步任务 |

### 1.2 状态更新约定

所有钩子返回 `stateUpdate map[string]interface{}`，遵循以下约定：

- 返回 `nil` 或空 `map`：不修改 state
- 返回非空 `map`：合并到 state（浅层合并）
- 错误返回 `err`：中断执行，向上透传

### 1.3 ThreadID 传递

通过 `context` 传递 `thread_id`，key 为：

```go
// context key 定义
type threadIDKey struct{}

// 设置 thread_id
ctx = context.WithValue(ctx, threadIDKey{}, "thread-123")

// 获取 thread_id
threadID, _ := ctx.Value(threadIDKey{}).(string)
```

```

```

```
1. ThreadDataMiddleware        (BeforeAgent)
2. LoopDetectionMiddleware    (AfterModel)
```

## 3. 各中间件详细规范

### 3.1 ThreadDataMiddleware

**DeerFlow 参考**：[thread_data_middleware.py](../deerflow-reference/thread_data_middleware.py)

#### 职责
- 创建线程目录结构
- 计算 workspace/uploads/outputs 路径
- 填充 `state.ThreadData`

#### State 读写
| 操作 | 字段 | 类型 |
|------|------|------|
| 写 | `thread_data` | `state.ThreadDataState` |
| 写 | `thread_data.workspace_path` | `string` |
| 写 | `thread_data.uploads_path` | `string` |
| 写 | `thread_data.outputs_path` | `string` |

#### 配置
| 字段 | 说明 | 默认值 |
|------|------|--------|
| `BaseDir` | 基础目录 | `pkg/config.Paths.BaseDir` |
| `LazyInit` | 是否懒加载 | `true` |

#### 实现要点
1. **路径计算**：使用 `pkg/config/paths.go` 的 `Paths.SandboxWorkDir(threadID)` 等
2. **懒加载**：`LazyInit=true` 时只计算路径，不创建目录
3. **目录结构**：
   ```
   {base_dir}/threads/{thread_id}/user-data/
   ├── workspace/   # 工作区
   ├── uploads/     # 上传文件
   └── outputs/     # 输出文件
   ```

#### 测试用例
- [ ] ThreadID 为空时返回错误
- [ ] LazyInit=true 时不创建目录
- [ ] LazyInit=false 时创建目录
- [ ] 路径正确填充到 state.ThreadData

### 3.2 UploadsMiddleware

**DeerFlow 参考**：[uploads_middleware.py](../deerflow-reference/uploads_middleware.py)

#### 职责
- 从最后一条 HumanMessage 的 `additional_kwargs.files` 提取新上传文件
- 扫描历史上传文件（uploads 目录）
- 生成 `<uploaded_files>` 消息块，注入到 HumanMessage 内容前

#### State 读写
| 操作 | 字段 | 类型 |
|------|------|------|
| 读 | `messages` | `[]schema.Message` |
| 写 | `messages` | `[]schema.Message` (修改最后一条) |
| 写 | `uploaded_files` | `[]state.UploadedFile` |

#### 实现要点
1. **新文件提取**：从 `schema.HumanMessage.AdditionalKwargs["files"]` 提取
2. **历史文件扫描**：遍历 `thread_data.uploads_path` 目录
3. **去重**：新文件和历史文件按 `filename` 去重
4. **消息注入**：`<uploaded_files>` 块注入到 HumanMessage 内容前
5. **保留 additional_kwargs**：修改消息时保留原始 `additional_kwargs`

#### `<uploaded_files>` 格式
```xml
<uploaded_files>
The following files were uploaded in this message:

- filename.txt (1.2 KB)
  Path: /mnt/user-data/uploads/filename.txt

The following files were uploaded in previous messages and are still available:

- old-file.json (0.5 KB)
  Path: /mnt/user-data/uploads/old-file.json

You can read these files using the `read_file` tool with the paths shown above.
</uploaded_files>
```

#### 测试用例
- [ ] 无文件时返回 nil
- [ ] 有新文件时正确提取
- [ ] 历史文件正确扫描
- [ ] `<uploaded_files>` 正确注入
- [ ] additional_kwargs 保留

### 3.3 SandboxMiddleware

**DeerFlow 参考**：`deer-flow/backend/packages/harness/deerflow/sandbox/middleware.py`

#### 职责
- 从 `SandboxProvider` acquire sandbox
- 存储 `sandbox_id` 到 state
- 在 cleanup 时 release sandbox

#### State 读写
| 操作 | 字段 | 类型 |
|------|------|------|
| 写 | `sandbox` | `state.SandboxState` |
| 写 | `sandbox.sandbox_id` | `string` |

#### 配置
| 字段 | 说明 |
|------|------|
| `SandboxProvider` | `sandbox.SandboxProvider` 接口 |

#### 实现要点
1. **Acquire**：调用 `provider.Acquire(ctx, threadID)`
2. **LazyInit**：只在首次需要时 acquire
3. **Release**：提供 `Release(ctx, threadID)` 方法（不在钩子中调用，由外部 cleanup 触发）

#### 测试用例
- [ ] SandboxProvider 为 nil 时使用 default
- [ ] SandboxID 正确存储
- [ ] 重复调用不重复 acquire
- [ ] Release 正确调用

### 3.4 DanglingToolCallMiddleware

**DeerFlow 参考**：[dangling_tool_call_middleware.py](../deerflow-reference/dangling_tool_call_middleware.py)

#### 职责
- 检测 AIMessage 有 `tool_calls` 但没有对应 ToolMessage
- 插入 synthetic ToolMessage，status="error"

#### State 读写
| 操作 | 字段 | 类型 |
|------|------|------|
| 读 | `messages` | `[]schema.Message` |
| 写 | `messages` | `[]schema.Message` (插入 ToolMessage) |

#### 实现要点
1. **检测挂起**：遍历 messages，匹配 `AIMessage.ToolCalls` 和 `ToolMessage.ToolCallID`
2. **合成消息**：对缺失的 ToolCall，插入：
   ```go
   schema.ToolMessage{
       ToolCallID: tc.ID,
       Content: "tool call interrupted",
       Status: "error",
   }
   ```
3. **实现接口**：`WrapModelCallMiddleware`（在模型调用前检查历史消息）

#### 测试用例
- [ ] 无挂起调用时不修改
- [ ] 有挂起调用时插入 synthetic ToolMessage
- [ ] 多个挂起调用全部补全

### 3.5 SummarizationMiddleware

**DeerFlow 参考**：`deer-flow/backend/packages/harness/deerflow/agents/lead_agent/agent.py` (SummarizationMiddleware 创建)

#### 职责
- 按 trigger 条件检测是否需要摘要
- 调用模型生成摘要
- 应用 keep 策略

#### 配置（来自 `pkg/config.SummarizationConfig`）
| 字段 | 说明 |
|------|------|
| `Enabled` | 是否启用 |
| `Trigger` | 触发条件 (`tokens`/`messages`/`fraction`) |
| `Keep` | 保留策略 (`last_n`/`first_n`/`fraction`) |
| `ModelName` | 摘要用模型 |
| `TrimTokensToSummarize` | 摘要前截断 token 数 |
| `SummaryPrompt` | 自定义摘要提示词 |

#### 实现要点
1. **触发检测**：检查消息数/token数是否达到 trigger
2. **模型调用**：使用轻量模型生成摘要
3. **Keep 策略**：
   - `keep.last_n`: 保留最后 N 条消息
   - `keep.first_n`: 保留最前 N 条消息
   - `keep.fraction`: 保留一定比例

#### 测试用例
- [ ] Enabled=false 时不执行
- [ ] Trigger 未达到时不执行
- [ ] Trigger 达到时生成摘要
- [ ] Keep 策略正确应用

### 3.8 MemoryMiddleware

**DeerFlow 参考**：[memory_middleware.py](../deerflow-reference/memory_middleware.py)

#### 职责
- 过滤消息：只保留 User 输入 + 最终 AI 响应
- 去抖队列：30s 默认
- 异步内存更新：不阻塞主流程

#### State 读写
| 操作 | 字段 | 类型 |
|------|------|------|
| 读 | `messages` | `[]schema.Message` |

#### 消息过滤规则
**保留**：
- HumanMessage（去掉 `<uploaded_files>` 块）
- AIMessage **无** tool_calls（最终响应）

**过滤**：
- ToolMessage（中间结果）
- AIMessage **有** tool_calls（中间步骤）
- 仅包含 `<uploaded_files>` 的 HumanMessage 及其配对 AI

#### 实现要点
1. **过滤**：`_filter_messages_for_memory()`
2. **去重**：同一 thread 30s 内多次更新只处理一次
3. **入队**：调用 `memory.GetQueue().Add(threadID, messages, agentName)`

#### 测试用例
- [ ] 工具消息正确过滤
- [ ] 中间 AI 响应正确过滤
- [ ] `<uploaded_files>` 块正确去除
- [ ] 消息正确入队

### 3.12 LoopDetectionMiddleware

**DeerFlow 参考**：[loop_detection_middleware.py](../deerflow-reference/loop_detection_middleware.py)

#### 职责
- 检测重复工具调用循环
- Warn threshold（默认 3）：注入警告消息
- Hard limit（默认 5）：强制停止，去掉 tool_calls

#### 配置
| 字段 | 说明 | 默认值 |
|------|------|--------|
| `WarnThreshold` | 警告阈值 | 3 |
| `HardLimit` | 强制停止阈值 | 5 |
| `WindowSize` | 滑动窗口大小 | 20 |
| `MaxTrackedThreads` | 最大跟踪线程数（LRU） | 100 |

#### 实现要点
1. **Order-independent 哈希**：
   - 对每个 tool call：`name` + JSON 序列化 `args` (sort_keys=true)
   - 按 name + args 哈希排序
   - 整体 JSON 序列化后 MD5（取前 12 字符）
2. **滑动窗口**：每线程最近 N 个哈希
3. **LRU 淘汰**：超过 `MaxTrackedThreads` 时淘汰最久未用
4. **警告去重**：每个哈希每线程只警告一次
5. **实现接口**：`AfterModelMiddleware`

#### 警告消息
```
[LOOP DETECTED] You are repeating the same tool calls.
Stop calling tools and produce your final answer now.
If you cannot complete the task, summarize what you accomplished so far.
```

#### 强制停止消息
```
[FORCED STOP] Repeated tool calls exceeded the safety limit.
Producing final answer with results collected so far.
```

#### 测试用例
- [ ] 无重复不触发
- [ ] 达到 WarnThreshold 注入警告
- [ ] 达到 HardLimit 去掉 tool_calls
- [ ] LRU 淘汰正确工作
- [ ] 警告去重正确
