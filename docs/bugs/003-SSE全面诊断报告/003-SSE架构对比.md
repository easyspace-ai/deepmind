## 1. 当前制系统流图（存在问题）

```
前端请求
    │
    ▼
┌──────────────────────────────────────────────────────┐
│  /api/langgraph/threads/{id}/runs/stream (POST)      │
│                streamRun()                            │
├──────────────────────────────────────────────────────┤
│  1. WriteMetadata()                   ✅             │
│     ├─ 立即发送 metadata 事件                        │
│     └─ 时间：0-100ms                                  │
│                                                      │
│  2. WriteMessagesTuple(userMsg)       ✅             │
│     ├─ 发送用户消息                                  │
│     └─ 时间：100-200ms                                │
│                                                      │
│  3. 启动 keep-alive goroutine         ⚠️              │
│     ├─ 每 5s 发送 "keep-alive" 注释                  │
│     └─ 但主goroutine仍被 recv() 阻塞 ❌              │
│                                                      │
│  4. ChatModelAdapter.Stream()         ❌              │
│     ├─ 实际上是 Generate() 同步调用                  │
│     ├─ 可能阻塞 30-120s                              │
│     ├─ keep-alive 无法完全救                        │
│     └─ 时间：200ms - 30s+                            │
│          ↓                                           │
│     [长期静默！keep-alive 在后台发送]                │
│                                                      │
│  5. WriteMessagesTuple(aiMsg delta)   ⚠️              │
│     ├─ 逐个token发送AI响应                           │
│     ├─ 但如果上游卡，可能一直卡                     │
│     └─ 时间：100ms-50s+                              │
│          ↓                                           │
│     [如果这里超时，流中断]                           │
│                                                      │
│  6. WriteValues(state)                ✅              │
│     ├─ 发送最终状态快照                              │
│     └─ 时间：50s+                                     │
│                                                      │
│  7. WriteEnd()                        ⚠️ 风险        │
│     ├─ 理论上：确保发送                              │
│     ├─ 实际上：某些错误分支缺失                      │
│     └─ 时间：不确定（可能缺失）                      │
│                                                      │
│  8. 返回                              ⚠️              │
│     ├─ 关闭 keep-alive goroutine                     │
│     ├─ 但 defer 之后 Writer.Close()                  │
│     └─ 客户端可能还在等待                            │
└──────────────────────────────────────────────────────┘
    │
    ↓
前端 @langchain/langgraph-sdk useStream hook
    │
    ├─ 事件1: metadata  ✅ 收到
    │
    ├─ 事件2: messages-tuple (user)  ✅ 收到
    │
    ├─ 事件3-N: keep-alive (注释) ⚠️  被SDK内部或浏览器吞掉
    │
    ├─ 事件N+1: messages-tuple (ai) 🤔 何时收到？
    │           如果 30s+ 还没来，前端可能超时
    │
    ├─ 事件最终: values 🤔 何时收到？
    │
    └─ 事件最后: end ⚠️ 可能缺失！
           ↓
    页面显示结果（不稳定）
```

## 2. 问题链条可视化

```
深层根本原因
│
├─ ChatModelAdapter.Stream() 非真流式
│  └─> Generate() 任意卡住（无token流动）
│      └─> streamReader.Recv() 持续阻塞
│          └─> keep-alive 虽然发送但无法唤醒主loop
│              └─> 客户端长期无业务事件
│                  └─> 前端超时 OR 代理超时
│                      └─> 连接中断
│                          └─> 流挂起或断开
│                              └─> 前面收到的metadata/messages-tuple无法完成
│                                  └─> 表现：用户看到部分消息然后卡住
│
└─ 错误路径未完整处理
   └─> 某些分支没有 writeStreamEnd()
       └─> 流没有明确的 "end" 事件
           └─> 客户端不知道流已结束
               └─> 继续等待（直到超时）
                   └─> 前端报"网络错误"
                       └─> 页面崩溃或冻结
```

```
时间    后端                             Goroutine             前端
────    ────                             ─────────             ────

1s      WriteMetadata() ──────────────► [metadata事件] ──────► 收到
        WriteMessagesTuple(user)
        ──────────────────────────────► [messages-tuple] ────► 收到
5s      │ Generate()阻塞中...           keep-alive: sent ────► 收到、忽略
        │                                                     （由SDK处理）
10s     │ Generate()仍阻塞...          keep-alive: sent ────► 收到、忽略
        │ [无任何业务事件] 
15s     │                              keep-alive: sent ────► 收到、忽略
        │ Generate()仍阻塞...
20s     ❌ 前端超时或代理断开             ↓                    等不及了，超时
        ⚠️ 或 Generate() 返回错误        keep-alive: sent      →connection reset
        但没有 writeStreamEnd()          ─────────────────X
        
30s     想要发送 values
        但连接已断开                                          显示"网络错误"
```

## 6. 关键改进点总结

```
【改进1】规范化结束路径
  ─────────────────────
  所有error分支都显式 writeStreamEnd()
  defer 作为最终保障（sync.Once防重复）
  ✓ 确保客户端一定收到 end 事件
  
【改进2】改进客户端断开检测
  ───────────────────────
  RecV() 用 select + timeout
  定期检查 context.Done()
  ✓ 客户端断开时响应<2s而不是>10s
  
【改进3】增加详细日志
  ───────────────────
  每个阶段记录时间戳
  error标记原因（metadata_error等）
  ✓ 方便排查具体卡在哪个阶段
  
【改进4】改进超时参数
  ───────────────
  总流：120s
  单chunk：30s无反应则重检查
  ✓ 对慢网络或响应慢的模型友好
  
【改进5】前端解耦SDK
  ────────────────
  创建 useDeerFlowStream() 
  直接面向SSE协议，不依赖SDK
  ✓ 便于调试、扩展、升级
```
