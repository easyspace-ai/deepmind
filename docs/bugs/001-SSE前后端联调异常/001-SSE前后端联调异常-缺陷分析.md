# SSE前后端联调异常 - 缺陷分析

## 1. 根因分析

### 1.1 后端写流并发

- keep-alive goroutine 与主消息循环并发写同一个 `ResponseWriter`
- 当前 `pkg/sse.Writer` 未提供串行化写保护
- 导致 SSE 事件可能交错，严重时会出现中断或浏览器侧解析异常

### 1.2 结束流程不一致

- `stopKeepAlive` 存在重复关闭风险
- 客户端断开、流结束、函数正常返回三条路径的结束逻辑未统一
- 容易出现 panic、重复结束事件或不完整响应

### 1.3 事件协议偏离

- 当前后端实际发送 `metadata`/`updates`/`finish`
- `frontend/` 主链路基于 LangGraph SDK，更适配 `messages-tuple`/`values`/`end`
- 协议偏移导致前端不得不保留历史兼容逻辑，增加联调复杂度

### 1.4 恢复能力错配

- 前端开启 `streamResumable`
- 后端 `joinStream` 尚未实现
- 导致前端能力声明和后端实际能力不一致

## 2. 修复策略

- 后端恢复标准 DeerFlow 事件名
- 去掉并发 keep-alive 写法，统一串行输出
- 收敛结束路径，确保只关闭一次、只结束一次
- 前端暂时关闭 `streamResumable`，避免假恢复
