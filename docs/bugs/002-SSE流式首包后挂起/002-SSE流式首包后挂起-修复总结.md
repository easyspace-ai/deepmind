# SSE 流式首包后挂起 - 修复总结

| 项目 | 内容 |
|------|------|
| 缺陷编号 | 002 |
| 创建日期 | 2026-03-25 |
| 状态 | 已修复 |

## 1. 修复内容

- 为 `pkg/sse.Writer.WriteComment()` 增加互斥保护，保证 keepalive 注释与业务事件串行写出
- 为 `pkg/sse/deerflow.Writer` 增加 `WriteKeepAlive()` 转发方法
- 在 `internal/api/langgraph_handler.go` 的 `streamRun()` 中恢复安全 keepalive
- 将 SSE 结束事件收敛为 `sync.Once` 单出口，避免重复 `end`
- 为模型调用增加请求级超时，降低长时间挂起导致的无结束流风险
- 新增后端回归测试，断言 fallback 场景下完整输出 `metadata`、`messages-tuple`、`values`、`end`
- 将 `internal/api/server.go` 的 `WriteTimeout` 从固定 `15s` 调整为 `0`，避免 HTTP 服务层主动切断 SSE 长连接
- 新增 `internal/api/server_test.go`，锁定 SSE 兼容的超时配置

## 2. 验证结果

- `env GOCACHE=/tmp/gocache go test ./internal/api ./pkg/sse ./pkg/sse/deerflow ./pkg/agent/provider` 通过
- 回归测试 `TestStreamRunFallbackCompletesSSESequence` 通过
- 回归测试 `TestNewServerDisablesWriteTimeoutForSSE` 通过
- 本次在代码层确认 SSE 不会只停留在首包 `metadata`
- 本次确认服务端不会再因为固定 `15s WriteTimeout` 主动切断流

## 3. 风险与后续

- `ChatModelAdapter.Stream()` 仍然不是逐 token 真流式，只是通过 keepalive 缓解长时间静默
- `joinStream` 仍未实现，前端恢复能力仍需谨慎开启
- 建议后续继续补一轮真实浏览器联调和上游模型慢响应压测
- 由于 `WriteTimeout` 已取消，SSE 请求生命周期更多依赖客户端断开、上游超时和应用层收尾逻辑
