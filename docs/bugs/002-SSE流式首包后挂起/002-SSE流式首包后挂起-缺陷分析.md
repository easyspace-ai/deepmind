# SSE 流式首包后挂起 - 缺陷分析

| 项目 | 内容 |
|------|------|
| 缺陷编号 | 002 |
| 创建日期 | 2026-03-25 |
| 状态 | 分析完成 |

## 1. 根因分析

### 1.1 HTTP 服务层对所有响应统一施加了 15 秒 WriteTimeout

`internal/api/server.go` 当前为整个 `http.Server` 配置了固定的：

- `ReadTimeout: 15s`
- `WriteTimeout: 15s`

SSE 属于长时间持续写出的响应，`WriteTimeout` 一旦到期，服务端会直接切断连接。由于当前日志里每次都稳定在 15 秒触发 `context canceled`，这说明首要根因并不是前端随机断开，而是服务端超时策略不适合 SSE。

### 1.2 流式接口在模型调用期间长期静默

当前 `streamRun` 在发送 `metadata` 后，会进入模型调用路径。项目里的 `ChatModelAdapter.Stream()` 并不是真正逐 token 下发，而是先同步执行 `Generate()`，再把完整消息塞进 `schema.StreamReader`。

这意味着：

- 模型尚未返回前，SSE 连接没有新的业务事件
- 前端或中间链路在长时间空闲后会主动取消请求
- 请求取消后，上游模型调用报 `context canceled`

### 1.3 现有 Writer 的注释写出没有并发保护

`pkg/sse.Writer.WriteComment()` 没有和 `WriteEvent()` 共用同一把锁。即使恢复 keepalive，如果 keepalive 注释与业务事件并发写入同一 `ResponseWriter`，仍可能造成流损坏或写出异常。

### 1.4 流结束路径不够集中

当前 `streamRun` 里多个分支都可能提前返回，容易出现：

- 没有发送 `end`
- 连接静默过久
- 客户端只能看到首包，无法明确得知流已结束

## 2. 修复方向

1. 取消 `http.Server` 对 SSE 的固定 15 秒 `WriteTimeout` 限制，避免服务端主动切断长连接
2. 为 `pkg/sse.Writer` 的注释写出补上互斥保护，保证 keepalive 与业务事件串行输出
3. 在 `streamRun` 中恢复安全 keepalive，在模型调用期间定时发送注释保活
4. 统一结束路径，确保成功、错误、客户端断开都能稳定收尾
5. 保留回退消息策略，避免上游失败时整个流只停留在 `metadata`
